# Contract: Dashboard Frontend App

## Purpose

The dashboard frontend app is an external browser client for the local dashboard API. Its only data boundary is HTTP JSON from spec 007. It must not import main project internals, parse local run files, or recompute analyzer facts.

## App Package Boundary

The app lives under:

```text
dashboard/
```

Required package properties:

- Own `package.json`.
- Own install lockfile after dependencies are installed.
- Own typecheck, test, build, preview, and dev commands.
- No dependency on root `src/` files.
- No TypeScript path alias that resolves to root `src/`.
- Dashboard-owned scripts for dev, preview, build, typecheck, test, import-boundary, package-boundary, contract-fixtures, style-boundary checks, and HTTP fixture capture.
- Browser API base URL configuration uses `VITE_DASHBOARD_API_BASE_URL`.

Forbidden source imports from `dashboard/src/**`:

```text
../src/*
../../src/*
@/../src/*
src/*
/Users/*/TokenEfficiencyTracker/src/*
```

Equivalent relative traversal into root `src/` is also forbidden.

## Supported API Contract Version

The app supports:

```ts
type SupportedSchemaVersion = 1;
```

Every successful response from the dashboard API must include:

```ts
type ApiEnvelope<T> = {
  schema_version: 1;
  generated_at: string;
  data: T;
  caveats: DashboardCaveat[];
};
```

If `schema_version` is absent or not supported, the app enters a version mismatch state and does not render the payload as valid dashboard data.

## Contract Fixture Boundary

The app duplicates API-facing types locally, so it must keep API-real fixture validation from the first implementation.

Required API-real fixtures:

```text
dashboard/scripts/capture-api-fixtures.ts
dashboard/test/fixtures/api-real/status.json
dashboard/test/fixtures/api-real/sessions.json
dashboard/test/fixtures/api-real/run.json
dashboard/test/fixtures/api-real/artifact-detail.json
dashboard/test/helpers/contract-fixtures.ts
```

Rules:

- API-real fixtures are captured over HTTP from the completed spec 007 dashboard API contract, not generated by importing root project code.
- `dashboard/src/**` must not import `dashboard/test/**`, fixture JSON, or fixture-loading helpers.
- Dashboard tests consume fixture JSON only and do not import root `src`.
- Fixture tests fail on missing or unsupported `schema_version`.
- Fixture tests fail when required envelope, status, session, run, artifact, privacy, caveat, or structured-error fields drift incompatibly.
- Hand-authored edge fixtures are allowed, but they do not replace API-real baseline fixtures.

## Client Endpoints

The app consumes these endpoints from the spec 007 contract:

- `GET /api/status`
- `GET /api/sessions?limit=20`
- `GET /api/runs/:runId`
- `GET /api/runs/:runId/artifacts/:artifactId`

The app may make repeat requests for freshness. It must not send write requests for captured run data.

Session identity rules:

- Use `session.run_id` as the routable identifier for `getRun(runId)` and `getArtifactDetail(runId, artifactId)`.
- Treat `session.canonical_run_id`, when present, as display or diagnostic metadata only.
- Persist selected session state by routable `run_id`, not by canonical run ID or display label.

## App-Side API Client

```ts
type DashboardApiClient = {
  getStatus(): Promise<StatusResponse>;
  getSessions(limit?: number): Promise<SessionsResponse>;
  getRun(runId: string): Promise<RunResponse>;
  getArtifactDetail(runId: string, artifactId: string): Promise<ArtifactDetailResponse>;
};
```

Rules:

- `baseUrl` is configurable for local development.
- The browser-facing environment variable is `VITE_DASHBOARD_API_BASE_URL`.
- Network failures are normalized into an offline state.
- API structured errors are preserved for display.
- Successful responses are version-checked before state updates.

## App Architecture Boundary

Required app layers:

- `App.tsx`: mount point only.
- `DashboardShell`: app layout and top-level status/offline/version state display.
- `DashboardController`: API status/session/run/detail orchestration, URL state, refresh lifecycle, and state reconciliation.
- `RunExplorer`: run-scoped overview, task, filter, table, caveat, privacy, and artifact detail composition.
- Small hooks for status, sessions, selected run, artifact detail, refresh, and URL state.
- Pure utilities for filtering, sorting, searchable text use, and refresh reconciliation.
- Central privacy display policy for hidden, metadata-only, unavailable, uncaptured, preview, and raw-available states.

Rules:

- `canonical_run_id` is never used for routing, URL state, API route construction, or selected-run state.
- Filtering and sorting operate on API-provided artifact rows and safe search text only.
- Components render privacy labels through the centralized privacy display policy.

## View State Contract

```ts
type DashboardViewState = {
  selectedRunId?: string;
  selectedTaskGroupId?: string;
  selectedArtifactId?: string;
  categoryFilter: string | "all";
  searchQuery: string;
  sortKey:
    | "display_name"
    | "display_category"
    | "total_exposure"
    | "repeated_exposure"
    | "inclusion_count"
    | "estimated_cached_input_tokens"
    | "estimated_uncached_input_tokens";
  sortDirection: "asc" | "desc";
  refreshMode: "manual" | "interval";
};
```

Rules:

- Refresh preserves selected IDs only when refreshed payloads still contain them.
- `selectedRunId` stores the API-routable `run_id` from the sessions response.
- Filtering and sorting use API-provided artifact rows and safe search text.
- Sort keys are app view-state keys that correspond to API artifact row fields; if future API metadata exposes default sort aliases, the API client must normalize them at the boundary before updating view state.
- The app does not derive new analyzer metrics from raw artifacts.

## Styling Boundary

Required dashboard-owned style modules:

```text
dashboard/src/styles/app.css

dashboard/src/styles/sessions.css
dashboard/src/styles/run-explorer.css

dashboard/src/styles/detail.css
```

Rules:

- `app.css` is an entrypoint or app-shell file, not a catch-all stylesheet.
- Root project styles and generated static dashboard styles are not imported.
- Privacy, caveat, loading, offline, empty, partial, stale, selected, focus, and disabled states use shared state/style vocabulary.
- Feature styles are owned by their dashboard area.

## Required UI States

The app must represent:

- Initial loading.
- API offline or unreachable.
- Status reachable but not ready.
- Unsupported response version.
- Empty sessions.
- Selected run loading.
- Selected run not found.
- Selected run partial or stale.
- Empty artifact results for active filters.
- Artifact detail loading.
- Artifact detail not found.
- Metadata-only or hidden content.
- Attribution caveats and estimated values.

## Contract Validation

Required validation scenarios:

- Status response gates app readiness and version compatibility.
- Session list renders safe labels, counts, availability, and caveats.
- Selected run renders overview, tasks, artifacts, filters, privacy state, and caveats.
- Artifact detail renders identity, metrics, metadata, relationships, privacy, and caveats.
- Offline, not-found, partial, stale, and version mismatch states are visible.
- Metadata-only fixtures render no hidden raw content.
- Large fixture filtering/sorting remains responsive for at least 1,000 artifact rows.
- Import-boundary tests fail on any `dashboard/src/**` import from root `src/` or test-owned `dashboard/test/**`.
- Package-boundary tests prove root source and root `tsconfig` do not include `dashboard/src`.
- Contract-fixture tests consume API-real fixtures.
- Style-boundary tests fail on root style imports or broad catch-all growth.
